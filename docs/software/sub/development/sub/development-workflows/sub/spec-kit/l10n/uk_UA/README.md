# GitHub Spec Kit

GitHub Spec Kit — open-source toolkit і CLI для Spec-Driven Development (SDD) з агентами для програмування на основі ШІ. Він перетворює задум продукту на довговічні специфікації, технічні плани, декомпозиції задач, артефакти валідації та workflows реалізації замість покладання на одноразові coding prompts.

## Огляд

Spec Kit реалізує процес розроблення, у якому специфікації залишаються першокласними артефактами проєкту. Coding agent проходить через явні етапи, а не самостійно виводить вимоги, архітектуру, порядок реалізації та критерії прийнятності з одного prompt.

Toolkit надає CLI `specify`, шаблони проєкту, скрипти, commands або skills для підтримуваних агентів і стандартний workspace `specs/` для артефактів feature.

Spec Kit сам не є coding agent, редактором коду чи універсальним оркестратором агентів. Це рівень development workflow навколо вибраного coding agent.

## Основний workflow

Типовий workflow використовує такі етапи:

1. **Ініціалізація інтеграції проєкту.** `specify init` установлює вибрану інтеграцію агента, шаблони й допоміжні скрипти.
2. **Визначення принципів проєкту.** `/speckit.constitution` фіксує довготривалі інженерні та governance-принципи.
3. **Специфікація feature.** `/speckit.specify` описує, що й навіщо слід створити, без передчасної фіксації реалізації.
4. **Уточнення неоднозначностей.** `/speckit.clarify` знаходить недостатньо визначені вимоги до технічного планування.
5. **Створення плану реалізації.** `/speckit.plan` фіксує архітектуру, вибір технологій, моделі даних, контракти, дослідження й експлуатаційні обмеження.
6. **Генерація виконуваних задач.** `/speckit.tasks` створює впорядковану декомпозицію задач із залежностями, шляхами, необов’язковою test-first роботою та позначками паралельності.
7. **Перевірка узгодженості.** `/speckit.analyze` і `/speckit.checklist` можуть перевіряти покриття, узгодженість та якість вимог до реалізації.
8. **Реалізація.** `/speckit.implement` просить вибраного coding agent виконати задачі згідно зі схваленими артефактами.
9. **Узгодження залишку роботи.** `/speckit.converge` може порівняти кодову базу зі специфікацією, планом і задачами, а потім додати невиконану роботу.

Додаткові commands можуть перетворювати задачі на GitHub issues або підтримувати встановлені extensions і власні workflows.

## Типові артефакти проєкту

Точна згенерована структура залежить від версії Spec Kit, інтеграції, увімкнених extensions і складності feature. Проєкт зазвичай містить:

```text
.specify/
  memory/
    constitution.md
  scripts/
    bash/
    powershell/
  templates/

specs/
  001-feature-name/
    spec.md
    plan.md
    tasks.md
    research.md
    data-model.md
    quickstart.md
    contracts/
```

Основні обов’язки артефактів:

- `constitution.md` — принципи й governance-обмеження всього проєкту;
- `spec.md` — сценарії користувача, вимоги, scope й очікування прийнятності;
- `plan.md` — архітектура реалізації та технічні рішення;
- `tasks.md` — впорядкована практична робота з реалізації;
- `research.md` — свідчення й рішення щодо невирішених технічних питань;
- `data-model.md` — сутності домену та зв’язки, коли доречно;
- `contracts/` — API-, protocol-, schema- або interface-контракти;
- `quickstart.md` — настанови з валідації чи виконання реалізованої feature.

Ці файли мають залишатися придатними до перегляду й контролю версій, а не зникати після генерації коду.

## Встановлення

Maintainers рекомендують установлювати позначений тегом release безпосередньо з офіційного GitHub-репозиторію за допомогою `uv`:

```bash
uv tool install specify-cli --from git+https://github.com/github/spec-kit.git@vX.Y.Z
```

Ініціалізуйте новий проєкт із явно вибраною інтеграцією:

```bash
specify init my-project --integration copilot
cd my-project
```

За потреби ініціалізуйте поточний репозиторій:

```bash
specify init . --integration copilot
```

Корисні commands обслуговування:

```bash
specify version
specify self check
specify self upgrade --dry-run
specify self upgrade
```

У production workflows використовуйте тег release, а не без змін приклад установлення з development branch.

Офіційна настанова зі встановлення попереджає, що пакети зі схожими назвами на PyPI не підтримуються проєктом Spec Kit. Установлюйте з офіційного GitHub-репозиторію або з локально зібраного пакета на його основі.

## Підтримка платформ та інтеграцій

Офіційний проєкт підтримує Linux, macOS і Windows. Поточна документація зі встановлення описує нативну підтримку PowerShell-скриптів у Windows без WSL; Bash-скрипти залишаються доступними для інших середовищ або явного вибору.

Spec Kit інтегрується з багатьма CLI- й IDE-based coding agents. Установлений CLI може показати інтеграції, підтримувані його версією:

```bash
specify integration list
```

Більшість інтеграцій надають commands `/speckit.*`. Інтеграції з підтримкою skills mode можуть установлювати відповідні agent skills замість slash-command prompt files.

Сумісність слід перевіряти для конкретних версій Spec Kit і coding agent. Задокументована інтеграція не гарантує однакової поведінки commands, дозволів інструментів, якості моделі чи ізоляції виконання між агентами.

## Extensions, presets і bundles

Spec Kit має кілька рівнів налаштування:

- **Локальні перевизначення проєкту** змінюють шаблони одного репозиторію.
- **Presets** налаштовують створення наявних workflows і артефактів Spec Kit.
- **Extensions** додають commands, шаблони, інтеграції або етапи розроблення.
- **Bundles** пакують версіонований, орієнтований на роль набір extensions, presets, steps і workflows.

Використовуйте preset, коли організація хоче змінити стандарти специфікації, планування чи задач. Використовуйте extension, коли потрібна справді нова можливість або етап workflow. Використовуйте bundle, коли повторюваний командний або рольовий набір слід надавати як одне ціле.

Налаштування підвищує гнучкість, але додає ще одну версіоновану поверхню конфігурації. Фіксуйте версії, перевіряйте встановлені компоненти й тестуйте оновлення до широкого застосування.

## Переваги

- Перетворює нечіткий задум feature на придатні до review артефакти до реалізації.
- Відокремлює вимоги продукту від архітектури та вибору реалізації.
- Полегшує перевірку припущень, обмежень, залежностей і критеріїв прийнятності.
- Підтримує кількох coding agents замість прив’язки workflow до одного провайдера.
- Заохочує поетапне уточнення й валідацію замість одноразової генерації.
- Зберігає специфікації та плани feature в системі контролю версій поряд із кодом.
- Підтримує greenfield-розроблення й ітераційну роботу з наявними кодовими базами.
- Надає точки розширення для стандартів і процесів організації.

## Компроміси й обмеження

- Workflow артефактів може бути надмірним для малих, очевидних і низькоризикових змін.
- Погані початкові вимоги все одно можуть створити докладні, але неправильні специфікації та плани.
- Згенеровані артефакти можуть розійтися з реалізацією, якщо команда не оновлює й не перевіряє їх.
- Більша кількість документів не гарантує кращої архітектури, безпеки чи продуктових рішень.
- Якість агента, обмеження контексту, доступ до інструментів і поведінка провайдера суттєво впливають на результати реалізації.
- `/speckit.implement` може спричинити суттєві зміни репозиторію й локального середовища через вибраного coding agent.
- Шаблони й commands розвиваються, тому команди мають фіксувати releases і перевіряти зміни оновлень.
- Spec Kit доповнює тести, code review, CI, безпекові засоби контролю й керування проєктом, а не замінює їх.

## Рекомендований підхід до впровадження

1. Випробуйте Spec Kit на обмеженій feature із чіткими критеріями прийнятності.
2. Запишіть принципи проєкту до генерації артефактів feature.
3. Виконайте уточнення до планування, коли вимоги неоднозначні.
4. Перевіряйте специфікацію та план як інженерні артефакти, за які відповідають люди.
5. Вимагайте явного схвалення до суттєвих кроків реалізації чи публікації.
6. Запускайте тести, linters, безпекові перевірки й звичайний code review після реалізації агентом.
7. Порівнюйте вартість підтримки артефактів зі зменшенням повторної роботи й неоднозначності.
8. Фіксуйте відомий release і спершу тестуйте оновлення в некритичному репозиторії.

## Контрольний список оцінювання

Оцінюйте toolkit у реальному середовищі розроблення:

- Чи має вибраний coding agent підтримувану інтеграцію?
- Чи повні й тестовані згенеровані специфікації та чи зрозумілі вони команді?
- Чи враховують плани наявну архітектуру й стандарти репозиторію?
- Чи достатньо малі задачі для review й відновлення після невдалого кроку реалізації?
- Чи може workflow зберігати межі людського схвалення для впливових змін?
- Чи працюють згенеровані скрипти й commands у потрібних операційних системах?
- Чи перевірено й зафіксовано версії extensions, presets і bundles?
- Чи можна простежити зміни специфікації до реалізації та тестів?
- Чи достатньо workflow покращує якість постачання, щоб виправдати додаткові артефакти?

## Відповідність AI Lab

Spec Kit належить до `development-workflows/`, бо його основна роль — структурувати розроблення ПЗ за допомогою ШІ. Він інтегрує coding agents, але переважно не діє як агент, редактор чи runtime оркестрації агентів.

Використовуйте цю сторінку як довідник для:

- керованих специфікаціями workflows coding agents;
- pipelines від вимог до реалізації;
- довговічних артефактів розроблення за допомогою ШІ;
- порівнянь із прямим prompting, самостійними coding agents і платформами оркестрації агентів;
- оцінювання governance і approval gates навколо згенерованого ШІ коду.
