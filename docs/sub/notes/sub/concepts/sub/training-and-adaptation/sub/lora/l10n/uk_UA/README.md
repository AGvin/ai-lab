# LoRA

Low-Rank Adaptation (LoRA) — параметроефективний метод fine-tuning, що навчає компактні низькорангові оновлення, зберігаючи початкові ваги моделі замороженими.

## Переклади

- [English](../../)
- Українська

## Основна ідея

Повний fine-tune оновлює значну частину параметрів моделі. Натомість LoRA представляє оновлення вибраних шарів як добуток двох набагато менших навчуваних матриць. Базова модель не змінюється, а adapter зберігає лише вивчену різницю.

Це дає змогу одній базовій моделі підтримувати кілька компактних варіантів для задач, доменів, персонажів або стилів.

## Як це працює

1. Виберіть цільові модулі, зазвичай attention- або projection-шари.
2. Заморозьте початкові ваги моделі.
3. Приєднайте навчувані низькорангові матриці до вибраних модулів.
4. Навчайте на dataset адаптації лише параметри adapter.
5. Зберігайте adapter окремо або об’єднайте його із сумісними базовими вагами для розгортання.

Важливі налаштування включають rank, scaling factor, цільові модулі, learning rate, dropout, якість dataset і тривалість навчання.

## Практичне застосування

- Адаптувати мовну модель до домену, формату output або повторюваної задачі.
- Додати візуальний стиль, предмет, персонажа чи концепцію продукту до моделі зображень.
- Підтримувати кілька малих варіантів без зберігання повної копії моделі для кожного.
- Поширювати артефакти адаптації, вимагаючи від користувачів окремо отримати сумісну базову модель.

## Приклад

Команда може зберігати одну універсальну модель для коду й навчити окремі LoRA adapters для внутрішнього framework, стилю документації та workflow review. Runtime завантажує базову модель один раз і активує adapter, потрібний для поточної задачі.

## Сумісність

LoRA adapter прив’язаний до припущень конфігурації навчання. Перевіряйте:

- точне сімейство й ревізію базової моделі;
- сумісність tokenizer або text encoder;
- назви цільових шарів та архітектуру;
- формат adapter і підтримку runtime;
- чи завантажує розгортання adapters динамічно, чи потребує об’єднаних ваг.

Не слід вважати, що adapter, створений для одного checkpoint, правильно працюватиме з іншою моделлю лише через подібну назву.

## Компроміси й обмеження

- Низькорангової місткості може бути недостатньо для великих змін поведінки.
- Погані навчальні дані можуть додати артефакти, регресії чи запам’ятовані помилки.
- Кілька adapters можуть непередбачувано взаємодіяти в поєднанні.
- Об’єднання спрощує розгортання, але унеможливлює незалежне перемикання adapters.
- LoRA змінює поведінку; це не надійна заміна RAG, коли факти мають бути актуальними чи відстежуваними.

## Практичний контрольний список

- Чи можна розв’язати проблему без навчання за допомогою prompting, інструментів, structured output або RAG?
- Чи записано й доступно точну базову модель?
- Чи розділено навчальні, валідаційні та holdout-приклади?
- Які цільові модулі й rank відповідають передбаченій зміні?
- Як adapter оцінюватиметься відносно незміненої базової моделі?
- Чи залишиться він окремим, поєднуватиметься з іншими або буде об’єднаний для розгортання?
