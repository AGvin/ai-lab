# Ідемпотентність

Ідемпотентну операцію можна повторити після першого успішного застосування без створення додаткових ненавмисних наслідків.

## Переклади

- [English](../../)
- Українська

## Основна ідея

Агентні робочі процеси часто повторюють дії після спливу часу очікування або неоднозначних збоїв. Без ідемпотентності повторна спроба може двічі надіслати той самий лист, створити дубльовані записи, двічі стягнути плату або неодноразово застосувати одну зміну репозиторію. Ідемпотентність зазвичай реалізують на рівні інструмента чи застосунку, а не за допомогою промптингу.

## Практичні методи

- Додавайте стабільний ключ ідемпотентності до кожної логічної дії.
- Доречно використовуйте операції upsert або compare-and-set.
- Записуйте завершені побічні ефекти у стійкому стані.
- Перевіряйте поточний зовнішній стан перед повторною спробою.
- Відокремлюйте планування лише для читання від операцій запису.

## Компроміси й обмеження

Ідемпотентність потребує сховища, проєктування ключів і чітких меж дій. Деякі операції неможливо зробити цілком ідемпотентними — натомість їм потрібна компенсація або ручне узгодження.

## Типові помилки

- Припускати, що після спливу часу очікування нічого не відбулося.
- Створювати новий ідентифікатор запиту для кожної повторної спроби.
- Повторювати весь робочий процес, коли не вдався лише один крок.
- Покладатися на пам’ять моделі про завершення дії.
