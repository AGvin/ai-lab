# Shannon

Автономний агент пентестингу вебзастосунків і API з відкритим кодом, який виконує white-box-тестування за допомогою ШІ.

## Огляд

Shannon — самостійний агент пентестингу на основі ШІ, який аналізує вихідний репозиторій, планує ймовірні шляхи атаки, взаємодіє із запущеним застосунком, виконує реальні експлойти та звітує про знахідки, які може підтвердити доказами працездатної експлуатації.

Його процес охоплює розвідку з урахуванням вихідного коду, розвідку запущеного застосунку, спеціалізований аналіз вразливостей, експлуатацію та створення Markdown-звіту. Поточний задокументований фокус охоплює ін’єкції, cross-site scripting, server-side request forgery, проблеми автентифікації та авторизації.

## Відповідність AI Lab

Shannon належить до `agents/`, оскільки канонічним об’єктом є самостійний автономний агент пентестингу.

Це корисний орієнтир для:

- агентного пентестингу;
- динамічного тестування безпеки застосунків з урахуванням вихідного коду;
- звітів із підтвердженням експлуатацією;
- багатоагентних процесів безпеки;
- ізоляції локальних агентів безпеки;
- порівняння агентів безпеки з відкритим кодом із комерційними AppSec-платформами.

## Доступ і функціональний розрив

Shannon Open Source можна безкоштовно запускати самостійно без обмежень за кількістю місць або сканувань. Користувач усе одно надає облікові дані провайдера моделей і оплачує відповідні витрати на виконання моделей.

Це не безкоштовний еквівалент повної [платформи Keygraph](../../../../../development/sub/application-security-tools/sub/keygraph/l10n/uk_UA/). До основних комерційних або платформних можливостей належать:

- black-box- і grey-box-пентестинг;
- повноцінний SAST із розбором коду та аналізом Code Property Graph;
- SCA з перевіркою досяжності, пошук секретів, перевірки інфраструктури й контейнерів;
- тестування безпеки бізнес-логіки;
- безперервне виконання та нативні CI/CD-gates;
- централізовані знахідки, відповідальні, SLA, панелі та синхронізація робочих процесів;
- автоматизоване виправлення з точковою перевіркою;
- керовані, self-hosted та air-gapped корпоративні засоби контролю.

Цей розрив має негативно впливати на рішення про впровадження, якщо такі можливості входять до реальних вимог. Не обирайте Shannon лише через відкритий код із наміром відкласти критичне покриття до завершення інтеграції.

## Ключове обмеження: немає black-box-тестування

Shannon Open Source потребує доступу до вихідного коду й виконує white-box-тестування. Він не забезпечує незалежної black-box-перевірки розгорнутого застосунку без контексту вихідного коду.

Це ключове обмеження для комплексного пентестингу, оскільки зовнішній атакувальник зазвичай не має доступу до репозиторію. Black-box-тестування може виявити зовнішньо доступну поведінку, помилки розгортання й маршрутизації, порушення меж автентифікації, відкриті сторонні поверхні та runtime-умови, які процес із доступом до коду може незалежно не підтвердити.

Використовуйте Shannon як цільовий white-box-інструмент, а не як доказ проведення повної оцінки безпеки з погляду атакувальника. Поєднуйте його з авторизованим black-box-інструментом або сервісом, коли таке покриття важливе.

## Зв’язок із Keygraph

Shannon є рушієм пентестингу в центрі комерційної платформи Keygraph. Репозиторій із відкритим кодом надає white-box-агент для самостійного запуску; Keygraph доповнює його ширшим аналізом, black-box-тестуванням, керуванням знахідками, виправленням, перевіркою, інтеграціями та корпоративною експлуатацією.

Цей зв’язок важливий для оцінювання впровадження, оскільки проєкт із відкритим кодом одночасно корисний сам по собі та є входом до значно ширшого платного продукту.

## Операційні примітки

- Рекомендований процес використовує `npx @keygraph/shannon` і запускає тимчасовий Docker worker.
- У задокументованому процесі цільовий репозиторій монтується лише для читання, але запущений цільовий застосунок усе одно може бути змінений унаслідок експлуатації.
- Windows підтримується через WSL2, а не нативно чи через Git Bash.
- Офіційна підтримка моделей зосереджена на Claude-сумісних провайдерах і задокументованих маршрутах хмарних провайдерів; альтернативні проксі або конфігурації моделей потребують окремої перевірки.
- Тривалість сканування й вартість моделі залежать від складності застосунку, охоплення та вибраного провайдера.

## Безпечне використання

Shannon не є пасивним сканером.

- Запускайте його лише проти систем, якими ви володієте або на тестування яких маєте явний письмовий дозвіл.
- Типово не спрямовуйте його на production-системи.
- Використовуйте одноразові локальні або staging-дані, резервні копії та план відновлення.
- Очікуйте створення облікових записів, надсилання форм, змін стану, вихідних запитів та інших побічних ефектів експлуатації.
- Використовуйте окремі тестові облікові дані та вузько обмежені ключі провайдера моделей.
- Не надавайте worker-середовищу доступ до не пов’язаних секретів, репозиторіїв і мереж.
- Не скануйте недовірені або ворожі репозиторії без додаткової ізоляції; вихідний вміст може намагатися виконати непрямий prompt injection проти агента.
- Перевіряйте кожну знахідку та proof of concept перед тим, як вважати її підтвердженим доказом безпеки.

## Примітки щодо оцінювання

Перед упровадженням оцініть:

- чи відповідає white-box-покриття реальній меті безпеки;
- який окремий інструмент або платний сервіс надаватиме потрібне black-box-покриття;
- підтримувані класи вразливостей і відомі прогалини покриття;
- сумісність із провайдерами моделей, вартість виконання моделей та відтворюваність;
- ізоляцію виконання, мережевий доступ, роботу з обліковими даними та відновлення стану цілі;
- якість звіту, докази експлуатації, хибнонегативні результати й потребу перевірки людиною;
- зобов’язання AGPL-3.0 для модифікованих або сервісних розгортань;
- чи створює шлях комерційного оновлення неприйнятне обмеження функцій або сукупну вартість.
