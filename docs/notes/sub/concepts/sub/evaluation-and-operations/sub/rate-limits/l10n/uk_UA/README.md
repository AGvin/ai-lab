# Обмеження частоти

Обмеження частоти обмежують кількість або обсяг запитів, дозволених протягом певного періоду.

## Переклади

- [English](../../)
- Українська

## Основна ідея

Обмеження захищають провайдерів і застосунки від перевантаження, неконтрольованих агентів, зловживань і неочікуваних витрат. Їх можна виражати як запити за хвилину, вхідні чи вихідні токени за хвилину, паралельні запити, дії інструментів або грошові бюджети.

## Практичне застосування

- Обмежувати споживання моделей та інструментів для користувача або tenant.
- Не допускати вичерпання квот циклами агентів.
- Захищати downstream-бази даних і API.
- Розділяти інтерактивний і пакетний трафік.
- Забезпечувати справедливе використання спільної інфраструктури.

## Належна практика

Повертайте чітку інформацію про повторну спробу, застосовуйте exponential backoff із jitter і ставте в чергу лише роботу, що залишатиметься корисною після очікування. Застосовуйте обмеження на кількох рівнях: користувача, workflow, провайдера моделі та чутливого інструмента.

## Компроміси й обмеження

Суворі обмеження можуть погіршити зручність під час законних сплесків. Черги згладжують навантаження, але збільшують затримку й можуть обробляти застарілі запити. Розподілене застосування потребує спільних лічильників або координованих gateways.

## Типові помилки

- Негайна повторна спроба після відповіді про обмеження частоти.
- Обмеження запитів без обмеження токенів або згенерованого output.
- Одна квота для критичних і некритичних навантажень.
- Дозвіл згенерованим моделлю циклам обходити обмеження користувача.
