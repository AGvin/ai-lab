# Резервні моделі

Резервні моделі — альтернативні моделі, які використовують, коли пріоритетна модель недоступна, перевищує обмеження політики, не проходить валідацію або не може виконати запит.

## Переклади

- [English](../../)
- Українська

## Основна ідея

Резервування — частина дизайну надійності, а не просто другий API key. Альтернатива може відрізнятися можливостями, розміром контексту, поведінкою схем, безпековою політикою, вартістю чи розташуванням даних, тому workflow має визначати, коли заміна прийнятна.

## Практичне застосування

- Продовжувати роботу сервісу під час збоїв провайдера.
- Після помилки переходити від малої моделі до потужнішої.
- Переходити на локальну модель, коли чутливі дані не можуть залишати середовище.
- Використовувати текстову резервну модель, якщо вхід не містить зображень.

## Компроміси й обмеження

Резервні моделі можуть давати неузгоджені відповіді й порушувати припущення, зроблені для основної моделі. Резервування між провайдерами може додавати інші умови приватності, резидентності чи ліцензування.

## Належна практика

Регулярно тестуйте резервні шляхи. Перевіряйте схеми й сумісність prompts. За потреби повідомляйте про погіршені можливості. Не спрямовуйте чутливі дані непомітно до менш довіреного провайдера.

## Типові помилки

- Налаштування резервної моделі, яку ніколи не тестували.
- Нескінченні повторні спроби до перемикання.
- Припущення про однаковий контекст і підтримку інструментів.
- Приховування постійних збоїв основної моделі успіхом резервної.
