# Повторні спроби

Повторні спроби знову виконують допустиму невдалу операцію за обмеженими правилами, призначеними для відновлення після тимчасових проблем.

## Переклади

- [English](../../)
- Українська

## Основна ідея

Політика повторних спроб має відрізняти тимчасові збої від постійних помилок. Мережевий тайм-аут, обмеження частоти або тимчасова недоступність сервісу можуть зникнути пізніше; некоректні аргументи, відмова в дозволі й непідтримувані операції зазвичай не зникнуть.

## Добра політика повторних спроб

- Обмежуйте максимальну кількість спроб.
- Використовуйте експоненційне збільшення затримки та джитер.
- Дотримуйтеся настанов постачальника `retry-after`.
- Повторюйте лише ідемпотентні або безпечно дедупліковані дії.
- Записуйте кожну спробу й остаточний результат.
- Ескалюйте проблему або змінюйте стратегію після вичерпання бюджету.

## Компроміси й обмеження

Повторні спроби збільшують затримку й можуть посилювати відмови, коли багато виконавців повторюють запити одночасно. Повторний виклик моделі може дати іншу відповідь, але не виправляє відсутніх даних або хибних припущень.

## Типові помилки

- Повторювати кожен виняток без класифікації.
- Негайно повторювати запити в тісному циклі.
- Повторювати неідемпотентний запис після неоднозначного тайм-ауту.
- Приховувати повторні збої від систем спостережуваності.
