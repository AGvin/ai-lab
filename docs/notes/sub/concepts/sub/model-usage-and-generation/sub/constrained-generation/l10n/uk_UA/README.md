# Обмежена генерація

Обмежена генерація визначає, які токени модель може видавати, щоб вихід відповідав граматиці, схемі, регулярній мові або набору дозволених значень.

## Переклади

- [English](../../)
- Українська

## Основна ідея

Замість прохання до моделі імітувати формат декодер маскує недопустимі варіанти токенів під час генерації. Це може гарантувати синтаксично коректний JSON, відповідність формальній граматиці або обмежити поле відомими значеннями. Обмеження контролює форму, а не фактичну точність чи коректність для бізнесу.

## Практичне застосування

- JSON або типізовані корисні навантаження API.
- SQL чи предметно-орієнтовані мови з визначеною граматикою.
- Класифікації з переліком значень.
- Конфігураційні файли, які мають надійно розбиратися.
- Аргументи інструментів, що перевірятимуться перед виконанням.

## Міркування щодо проєктування

Узгоджуйте граматику з токенізатором і реалізацією середовища виконання. Уникайте схем із надмірною вкладеністю чи неоднозначними альтернативами. Після декодування застосовуйте семантичну перевірку, адже структурно коректні дата, шлях або ідентифікатор усе одно можуть бути недійсними чи небезпечними.

## Компроміси й обмеження

Обмеження зменшують помилки форматування, але можуть уповільнювати декодування або змушувати модель заповнювати дозволену структуру навіть за браку інформації. Надто суворі граматики також можуть не дозволити корисні пояснення чи поля невизначеності.

## Типові помилки

- Вважати коректний синтаксис перевіреним змістом.
- Дозволяти необмежені рядки в полях, які потім стають командами чи шляхами.
- Використовувати настільки складну схему, що модель не може надійно її заповнити.
- Не передбачати резервної поведінки, коли генерація не задовольняє обмеження.
