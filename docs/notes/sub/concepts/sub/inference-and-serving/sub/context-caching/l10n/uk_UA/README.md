# Кешування контексту

Кешування контексту повторно використовує раніше оброблені префікси промпту, щоб повторні запити не обчислювали ті самі вхідні токени заново.

## Переклади

- [English](../../)
- Українська

## Основна ідея

Багато запитів мають сталий префікс: системний промпт, визначення інструментів, довгий документ або історію розмови. Середовище чи постачальник може зберегти оброблене подання й продовжити з нього, коли наступний запит має ідентичний або сумісний префікс.

## Практичне застосування

- Повторні запити до того самого документа.
- Довгі системні промпти й схеми інструментів.
- Багатоходові розмови.
- Пакетні задачі зі спільним префіксом інструкцій.
- Агентні процеси, що повертаються до сталого контексту.

## Компроміси й обмеження

Кешування зменшує затримку й вартість оброблення промпту, але займає пам’ять або сховище. Ключі кешу мають враховувати точну модель, токенізатор, байти промпту, конфігурацію, а іноді й позицію. Невеликі зміни можуть інвалідувати кеш.

## Міркування щодо безпеки

Записи кешу мають бути ізольовані між користувачами й орендарями. Для чутливих промптів потрібні належні політики зберігання та шифрування. Влучення в кеш ніколи не повинно розкривати контекст іншого користувача.

## Типові помилки

- Плутати кешування промпту постачальником із KV-кешем активного запиту.
- Припускати, що семантично подібний текст дасть влучення в кеш.
- Повторно використовувати кешований стан після зміни моделі чи системного промпту.
- Ігнорувати інвалідацію кешу й межі приватності.
