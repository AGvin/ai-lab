# Поділ на фрагменти

Chunking ділить вихідний матеріал на менші придатні до retrieval одиниці, які можна індексувати, ранжувати й додавати до контексту моделі.

## Переклади

- [English](../../)
- Українська

## Основна ідея

Chunk має бути достатньо малим для точного retrieval, але достатньо великим для збереження інформації, потрібної для розуміння. Належні межі зазвичай відповідають структурі документа — заголовкам, абзацам, функціям, рядкам таблиць або реплікам розмови — замість довільної кількості символів.

## Поширені стратегії

- Фіксовані token windows із перекриттям.
- Сегментація за абзацами або заголовками.
- Рекурсивний поділ із переходом до менших одиниць.
- Поділ коду за функцією, класом або модулем.
- Батьківсько-дочірній retrieval, де малі chunks знаходять більший блок контексту.

## Практичне застосування

Вибирайте розміри chunks за tokenizer і бюджетом контексту downstream-моделі. Зберігайте metadata джерела, заголовки, номери сторінок і версії документів. Тестуйте retrieval реалістичними запитаннями, включно з тими, чия відповідь перетинає межу.

## Компроміси й обмеження

Малі chunks підвищують точність, але можуть втрачати контекст. Великі зберігають контекст, але знижують специфічність ранжування й споживають більше токенів. Надмірне перекриття збільшує сховище й може повертати дубльовані свідчення.

## Типові помилки

- Поділ таблиць, блоків коду чи визначень посередині.
- Один розмір chunk для кожного типу вмісту.
- Вилучення заголовків та ієрархії з metadata chunk.
- Оцінювання лише наявності ключових слів, а не здатності chunk відповісти на запит.
