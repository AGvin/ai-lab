# RAG (Retrieval-Augmented Generation)

Retrieval-Augmented Generation (RAG) — архітектура, що надає мовній моделі релевантну зовнішню інформацію до генерації відповіді.

## Переклади

- [English](../../)
- Українська

## Призначення

Ця сторінка пояснює RAG як практичний шаблон для обґрунтування відповідей великих мовних моделей приватними, актуальними чи доменними даними без повторного навчання моделі.

## Що це

RAG поєднує дві операції:

1. Отримати релевантну інформацію із зовнішнього джерела знань.
2. Згенерувати відповідь, використовуючи отриману інформацію як контекст моделі.

```text
LLM + retrieval із зовнішніх даних = RAG
```

RAG — не окремий тип моделі. Це системна архітектура навколо мовної моделі, одного чи кількох джерел даних і retrieval pipeline.

## Як це працює

Типова RAG-система має потік індексування та потік запиту.

### Потік індексування

```text
Документи
    -> parsing і нормалізація
    -> chunking
    -> embeddings і metadata
    -> пошуковий індекс
```

Документи поділяються на менші одиниці — chunks. Кожний chunk зберігається з корисними metadata: джерелом, назвою, розділом, шляхом, версією чи політикою доступу.

Багато систем також створюють embedding для кожного chunk. Embedding — числове представлення для пошуку семантично подібного вмісту.

### Потік запиту

```text
Запитання користувача
    -> аналіз запиту
    -> retrieval
    -> необов’язкове переранжування
    -> контекст prompt
    -> LLM
    -> обґрунтована відповідь
```

Retriever шукає кандидатні chunks. Необов’язковий reranker точніше впорядковує кандидатів до додавання найкращих результатів у контекст моделі.

## Основні компоненти

| Компонент | Відповідальність |
| --- | --- |
| Джерело даних | Зберігає оригінальні документи або записи. |
| Loader або parser | Виділяє й нормалізує текст, структуру та metadata. |
| Chunker | Ділить великі документи на придатні до retrieval одиниці. |
| Embedding-модель | Перетворює текст на векторні представлення для семантичного пошуку. |
| Пошуковий індекс | Зберігає придатний до пошуку текст, вектори й metadata. |
| Retriever | Вибирає кандидатні chunks для запиту. |
| Reranker | Змінює порядок кандидатів за оціненою релевантністю. |
| Generator | Створює кінцеву відповідь із запитання й отриманого контексту. |

Поширені пошукові індекси включають PostgreSQL із `pgvector`, Qdrant, Milvus, Weaviate, Pinecone, Elasticsearch, OpenSearch і Chroma.

## Стратегії retrieval

### Векторний пошук

Векторний пошук корисний, коли запит і релевантний текст виражають ту саму ідею різними словами.

Наприклад:

```text
Запит: Як зменшити використання пам’яті моделлю?
Документ: Чотирибітне квантування знижує вимоги до VRAM.
```

### Пошук за ключовими словами

Пошук за ключовими словами або BM25 часто кращий для точних ідентифікаторів, як-от:

- назви моделей;
- назви функцій;
- номери версій;
- ключі конфігурації;
- повідомлення про помилки;
- ідентифікатори продуктів.

### Гібридний пошук

Практичні RAG-системи часто поєднують семантичний retrieval і retrieval за ключовими словами:

```text
векторний пошук + пошук за ключовими словами + переранжування
```

Це покращує recall для запитань природною мовою, зберігаючи precision для точних термінів.

## Практичний приклад

Припустімо, AI Lab містить нотатки про локальні моделі, квантування, GPUs і inference runtimes.

Користувач запитує:

```text
Яку модель можна запустити на GPU з 12 GB VRAM?
```

RAG pipeline може отримати:

- вимоги моделей до пам’яті;
- нотатки про квантування;
- відомості про сумісність runtime;
- результати локальних benchmarks;
- висновки щодо конкретного обладнання.

Потім мовна модель відповідає за цими отриманими нотатками, а не покладається лише на навчальні дані.

## Мінімальний потік реалізації

```js
const queryEmbedding = await embeddingModel.embed(question);

const candidates = await vectorStore.search({
    vector: queryEmbedding,
    limit: 20
});

const rankedChunks = await reranker.rank({
    query: question,
    documents: candidates
});

const context = rankedChunks
    .slice(0, 5)
    .map((chunk) => chunk.content)
    .join('\n\n');

const answer = await llm.generate({
    system: [
        'Відповідай за наданим контекстом.',
        'Не вигадуй фактів, відсутніх у джерелах.',
        'Чітко повідомляй, коли доступних свідчень недостатньо.'
    ].join(' '),
    prompt: `Контекст:\n${context}\n\nЗапитання:\n${question}`
});
```

У цьому прикладі пропущено production-вимоги: контроль доступу, кешування, спостережуваність, цитування джерел, захист від prompt injection і синхронізацію індексу.

## Порівняння RAG зі спорідненими підходами

| Підхід | Основне призначення | Змінює ваги моделі | Найкраще підходить для |
| --- | --- | --- | --- |
| RAG | Надання зовнішніх знань під час запиту | Ні | Актуальної, приватної чи часто змінюваної інформації |
| Fine-tuning | Адаптація поведінки моделі або продуктивності задачі | Так | Стилю, формату output, доменної поведінки чи спеціалізованих задач |
| Prompt engineering | Спрямування поведінки інструкціями | Ні | Обмежень відповіді й формулювання задачі |
| Long-context prompting | Передавання вихідного матеріалу безпосередньо моделі | Ні | Малих або обмежених наборів документів, що поміщаються у вікно контексту |

Fine-tuning не є надійною заміною придатної до пошуку бази фактичних знань. RAG зазвичай легше оновлювати, бо документи можна повторно індексувати без повторного навчання моделі.

## Переваги

- Використовує приватну чи організаційну інформацію.
- Підтримує інформацію, що змінюється після навчання моделі.
- Дає змогу атрибутувати джерела й переглядати свідчення.
- Не потребує повторного навчання для звичайних оновлень знань.
- Може зменшувати кількість непідтверджених відповідей, коли retrieval повертає релевантні свідчення.
- Може працювати з хостинговими або локальними мовними моделями.
- Може забезпечувати контроль доступу на рівні документів до потрапляння контексту в модель.

## Обмеження

RAG не гарантує правильної відповіді.

Типові режими відмов включають:

- потрібне джерело ніколи не індексували;
- межі chunks вилучили важливий контекст;
- retrieval повернув нерелевантних кандидатів;
- правильний результат не потрапив до вибраного набору `top-k`;
- отримано застарілі або суперечливі документи;
- модель проігнорувала чи неправильно витлумачила наданий контекст;
- отриманий вміст містив prompt injection або недовірені інструкції;
- кінцева відповідь зробила твердження, не підтверджені отриманими свідченнями.

Потужна мовна модель не може надійно компенсувати слабкий retrieval pipeline.

## RAG для вихідного коду

Retrieval коду часто потребує більше, ніж універсальні embeddings тексту.

Корисні сигнали включають:

- шляхи файлів;
- symbols і signatures;
- класи й функції;
- imports і exports;
- abstract syntax trees;
- dependency graphs;
- пошук точних рядків;
- історію Git і metadata версій.

Запит на зразок `Де визначено createEmbedding?` зазвичай краще обслуговує пошук symbols або ключових слів, ніж сама векторна подібність.

## Оцінювання

Оцінюйте retrieval і генерацію окремо.

### Оцінювання retrieval

Вимірюйте, чи повертає система потрібні свідчення:

- recall at `k`;
- precision at `k`;
- mean reciprocal rank;
- normalized discounted cumulative gain;
- якість reranker;
- затримку retrieval.

### Оцінювання відповіді

Вимірюйте, чи є кінцева відповідь корисною й підтвердженою:

- фактичну правильність;
- вірність отриманим свідченням;
- точність цитування;
- повноту;
- відмову за недостатності свідчень;
- затримку й вартість відповіді.

Наскрізна якість відповіді може приховувати збої retrieval, тому обидва рівні слід тестувати незалежно.

## Поширені сценарії використання

- Assistants документації.
- Пошук у репозиторіях і вихідному коді.
- Внутрішні корпоративні бази знань.
- Assistants підтримки клієнтів.
- Аналіз контрактів і політик.
- Research assistants із цитуванням джерел.
- Каталоги продуктів і технічна підтримка.
- Персональні бази знань.

## Підсумок

RAG під’єднує мовну модель до зовнішніх знань під час запиту.

```text
запитання користувача
    + релевантні отримані свідчення
    + мовна модель
    = відповідь, обґрунтована контекстом
```

Його ефективність залежить від якості даних, структури документів, retrieval, переранжування, побудови prompt і оцінювання відповіді, а не лише від мовної моделі.
